방법론

ProposalNet 해부

BEV Encoder + Object Detection + Trajectory candidate의 역할
ProposalNet의 일은 '많은 trajectory를 만들고 fine-grained reasoning이 다룰 만한 후보만 추리는 것'입니다. 세 부품을 봅니다. (1) BEV Encoder — ResNet-34로 뽑은 이미지 feature와 SECOND로 voxel-encode한 LiDAR feature를, BEVFormer-M의 learnable BEV query가 deformable attention으로 fusion. 거기에 이전 시점들의 BEV feature를 concat하고 conv를 통과시켜 spatio-temporal $F_{BEV}$ 를 만듭니다. 논문은 perception range를 전방 [0, 64]m, BEV 해상도 0.125m 로 확장 — fine-grained safety는 멀리 있는 위험까지 봐야 의미가 있기 때문이죠. 또 현재 + 과거 2 frame 입력으로 motion prediction의 안정성을 확보합니다. (2) Object Detection — learnable object query가 $F_{BEV}$ 에서 deformable attention으로 instance representation을 뽑아 centroid·heading·class·motion을 예측. classification score threshold 아래는 버리고, 남은 instance query + 예측 trajectory가 SWNet으로 전달됩니다. 이때 detection은 단순 인지가 아니라 'reasoning의 입력 토큰'이라는 점이 중요 — detection 품질이 그대로 PwNC 정확도로 번집니다. (3) Trajectory candidate — 학습셋 ego trajectory에 K-means로 $K=256$ anchor를 미리 클러스터링. 각 anchor를 MLP로 임베딩해 planning query $Q_{plan}$ 를 만들고, trajectory-guided deformable attention 이 anchor trajectory를 따라 $F_{BEV}$ 를 샘플링해 시간적으로 정합된 feature를 모읍니다: $\hat{Q}_{plan} = \mathrm{FFN}(\mathrm{Deform\text{-}Attn}_{traj}(Q_{plan}, A, F_{BEV}))$. 이후 MLP로 trajectory 보정 + imitation score + 5개 scene-level 점수를 뽑아 weighted log-sum으로 $K'=128$ safety-aware candidate를 추립니다. 포인트: 이 단계는 아직 instance-level reasoning을 하지 않습니다. 'reasoning이 의미 있는 trajectory만 남기는' 필터 역할. 내일 SWNet이 이 128개 위에 sparse world를 띄웁니다.

Trigger Type B
Trajectory-guided Deformable Attention
planning query를 BEV feature와 결합할 때, query를 BEV 전체에 균등 attention하지 않고 **trajectory를 따라가는 reference point** 에서만 deformable sampling. 식: $\hat{Q}_{plan} = \mathrm{FFN}(\mathrm{Deform\text{-}Attn}_{traj}(Q_{plan}, A, F_{BEV}))$. 직관: '이 anchor가 가는 길의 BEV feature만 보자'. 효과는 두 가지 — (1) trajectory의 spatio-temporal 정합성이 query에 직접 들어감, (2) BEV 전체 attention 대비 계산량 감소. SWNet/FRNet에서도 동일한 메커니즘이 ego·surrounding agent의 미래 trajectory를 따라 재사용되어 일관된 sampling primitive로 작동. Ablation에서 DiffusionDrive head를 이 메커니즘으로 교체했을 때 PDMS +0.5.
K-means Trajectory Anchor
ego trajectory를 학습 데이터에서 모은 뒤 K-means로 $K=256$ 개 대표 trajectory로 클러스터링한 것. 각 anchor가 MLP를 거쳐 planning query의 초깃값을 결정. 직관: 'data-driven prior로 가능한 미래 경로의 manifold를 미리 구성'. 이는 anchor-free regression이 겪는 mode-collapse와 multi-modal 표현 부족을 우회하는 표준 트릭. SafeDrive에서는 이 anchor가 단순 후보가 아니라, sparse world 시뮬레이션의 conditioning 변수로도 사용되어 anchor의 다양성이 곧 world의 다양성을 결정. 함정: anchor 수($K$)가 너무 적으면 manifold 커버리지가 떨어지고, 너무 많으면 ProposalNet 단계 비용이 증가.

객관식 Q1
Trajectory-guided deformable attention이 일반 deformable attention과 다르게 가져오는 핵심 이점은?
trajectory-guided 변형은 reference를 trajectory에 묶어 시간적 정합성을 강제하고, BEV 전체 attention 대비 cost도 줄임. 동일 메커니즘이 SWNet·FRNet에서 재사용됨.
객관식 Q2
ProposalNet에서 perception range를 전방 [0, 64]m, BEV 해상도 0.125m로 확장한 의도는?
fine-grained safety는 멀리 있는 위험과 경계 부근 정밀도가 모두 필요. 64m 전방 + 0.125m 해상도는 PwNC·TwDAC 모두를 위한 perception 측의 사전 작업.
주관식
본인의 detection backbone이 ResNet-34 + SECOND 수준이라면, SafeDrive식 sparse world reasoning을 그대로 얹었을 때 어떤 perception 약점이 가장 먼저 reasoning 결과에 새어 나올 것 같습니까?
힌트
Sparse world는 instance query에 의존합니다. detection이 놓치는 카테고리(예: 자전거, 휠체어)와 거리(원거리 보행자)는 무엇인지 떠올려 보세요.
꼭 짚고 가야 할 것
Sparse world는 perception을 forward-propagate한 reasoning이므로, missed detection은 곧 missed risk가 된다. SafeDrive를 도입할 때는 backbone 업그레이드와 reasoning 모듈 추가가 분리될 수 없는 한 쌍이다.
