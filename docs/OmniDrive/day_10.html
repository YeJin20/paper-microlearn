Dual-System AD에 끌어다 쓰기

System 2 reasoning trace 데이터·스위칭 게이트 신호로의 활용
이제 본인 연구 맥락으로 가져와 봅시다. dual-system AD — 빠른 System 1(DiffusionDrive·SparseDrive류 E2E policy)과 느린 System 2(AutoVLA류 VLA reasoning)가 동적으로 전환되는 구조 — 에서 OmniDrive는 어디에 꽂힐까요? 가장 자연스러운 자리는 System 2 supervision 보강입니다. AutoVLA가 "왜 지금 멈춰야 하는가"를 자연어로 풀어쓰게 만들려면, 단일 전문가 궤적만으론 부족해요. OmniDrive의 counterfactual Q&A는 "가속했다면 충돌, 좌회전했다면 신호 위반" 같은 대안 결과를 언어로 제공합니다. dual-system 스위처가 System 2로 hand-off할 때 학습시킬 reasoning trace 데이터로 거의 그대로 쓸 수 있어요. 두 번째 각도는 스위칭 게이트의 trigger 신호. 반사실 체크리스트(충돌·신호 위반·drivable area)는 System 1이 자신 없는 시나리오를 식별하는 입력으로 재활용 가능합니다. close object 정의(3초·10m)는 그대로 risk gate의 feature가 되고, lane-change·U-turn 같은 high-level decision은 mode-switch 라벨로 활용 가능. 후속 작업으로 DriveVLM, Hydra-MDP, VADv2가 closed-loop·multi-target distillation으로 확장 중입니다. OmniDrive가 명시한 한계(다른 에이전트 reactive 거동 부재)는 그쪽 closed-loop 시뮬레이터에서 보완되는 흐름이에요. 머릿속 시도: 본인의 스위처에서 OmniDrive Q&A 중 어떤 항목을 게이트 결정 feature로 가장 먼저 쓰시겠어요?

향후 작업·로드맵을 단정할 때
객관식 Q1
dual-system AD 관점에서 OmniDrive를 활용하는 가장 자연스러운 자리는?
counterfactual Q&A는 'reasoning을 언어화한 supervision'이고, 체크리스트는 게이트 trigger로 재활용 가능해 System 2 쪽과 hand-off 신호 양쪽에 맞는다.
객관식 Q2
OmniDrive의 한계로 논문이 직접 명시한 것은?
Limitations 섹션이 'does not yet consider reactions from other agents'라고 명시한다.
주관식
본인의 dual-system 스위처에서 OmniDrive의 어떤 신호를 게이트 결정의 1순위 feature로 쓰시겠어요? 그리고 그 선택이 '다른 에이전트의 reactive 거동을 무시한다'는 한계와 어떻게 충돌할까요?
힌트
반사실 체크리스트는 'ego 궤적이 잘못된 경우'만 시뮬레이션합니다. 다른 차가 갑자기 멈추는 시나리오는 빠져 있다는 점을 짚어 보세요.
꼭 짚고 가야 할 것
OmniDrive의 풍부한 supervision은 'ego가 만든 결과'에 한정됩니다. dual-system 게이트가 진짜로 필요한 순간(타인의 의외 행동)에는 reactive 시나리오를 별도로 채워줄 closed-loop·multi-agent 데이터가 추가로 필요해요.
